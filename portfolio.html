            <h2 class="text-2xl font-bold text-gray-800 mb-4">Credit Card Fraud Detection: Detailed Approach</h2>

            <p class="mb-4">Alright, let's break down this credit card fraud detection project step by step, just like I'm talking you through it.</p>

            <h3>1. Project Overview</h3>

            <p>First off, the whole point of this project is to build a system that can spot fraudulent credit card transactions. We're working with a dataset that has a ton of regular transactions and a much smaller number of fraudulent ones – it's pretty imbalanced, which is a common challenge in these kinds of problems.</p>

            <h3>2. Data Pre-processing and Exploration</h3>

            <ul>

                <li>We started by bringing in all the necessary tools: <code>numpy</code> and <code>pandas</code> for handling the data, <code>matplotlib</code> and <code>seaborn</code>, and then some really important stuff from <code>sklearn</code> for splitting our data, running our model, and checking how well it performed.</li>

                <li>We loaded up our credit card data from a CSV file, called <code>creditcard.csv</code>.</li>

                <li>Then, we took a quick peek at the data using <code>head()</code> and <code>tail()</code>. A crucial step was using <code>info()</code> to see the data types and if there were any missing pieces. Turns out, some columns had one missing value each. We confirmed this with <code>isnull().sum()</code>.</li>

                <li>Since we can't have missing data, we just dropped those rows using <code>dropna()</code>. It's a quick fix for this small number of missing values.</li>

                <li>Next, we checked the 'Class' column to see how many transactions were legitimate (0.0) and how many were fraudulent (1.0). It was super imbalanced: 11906 legitimate vs. only 52 fraudulent.</li>

                <li>We also looked at the 'Amount' field and grouped the data by 'Class' and took the mean of all columns, to get a good sense of how the average values of the features differed between legitimate and fraudulent transactions.</li>

            </ul>

            <h3>3. Addressing Imbalanced Data (Under-sampling)</h3>

            <p>Because of that huge imbalance, we used a technique called **under-sampling**. We took a random sample of 52 legitimate transactions – the same number as our fraudulent ones. Then, we combined this smaller set with all the fraudulent ones to create a new, balanced dataset. Now it's perfectly balanced: 52 legitimate and 52 fraudulent transactions.</p>

            <h3>4. Model Training</h3>

            <ul>

                <li>Now that our data was prepped, we split it into features (x) and our target (y).</li>

                <li>We further split these into training (90%) and testing (10%) sets. We used <code>stratify=y</code> to make sure the proportion of fraud vs. non-fraud was the same in both sets.</li>

                <li>For our model, we chose **Logistic Regression**. We initialized the model and then fit it to our training data. A "ConvergenceWarning" popped up, which often points to the need for scaling the data.</li>

            </ul>

            <h3>5. Model Evaluation</h3>

            <ul>

                <li>We made predictions on our training data and got an accuracy of about **98.92%**.</li>

                <li>More importantly, we made predictions on our test data (data the model has never seen before) and got an accuracy of about **81.82%**.</li>

            </ul>

            <h3>6. Insights and Potential Improvements</h3>

            <p>The under-sampling helped, but the accuracy drop on test data hints at some overfitting. That warning is also a clue.</p>

            <ul>

                <li><strong>Data Scaling:</strong> This should help the Logistic Regression model converge better and potentially improve its performance.</li>

                <li><strong>Other Algorithms:</strong> Maybe a Random Forest or Gradient Boosting would perform better.</li>

                <li><strong>Look beyond accuracy:</strong> For fraud detection, **precision** and **recall** are often more important. The F1-score and AUC-ROC are also great metrics.</li>

                <li><strong>Consider over-sampling (like SMOTE):</strong> Instead of throwing away data, we could generate synthetic fraud examples to balance the dataset.</li>

                <li><strong>Hyperparameter tuning:</strong> We could play around with the model's settings to squeeze out more performance.</li>

                <li><strong>Cross-validation:</strong> This would give us a more stable and reliable estimate of how well our model generalizes.</li>

            </ul>

            <h2 class="text-2xl font-bold text-gray-800 mb-4">Customer Segmentation: Detailed Approach</h2>

            <p class="mb-4">The <code>Mall_Customers.csv</code> file contains customer data from a mall, including demographics like Gender and Age, along with Annual Income and a Spending Score. This dataset is likely used to understand different customer groups based on their purchasing behavior.</p>

            <h3>1. Project Overview</h3>

            <p>The main goal of this project is to divide mall customers into distinct groups, or "segments," based on their purchasing habits and demographic information. This kind of segmentation helps businesses understand their customer base better, allowing for more targeted marketing and personalized strategies. We'll be using a clustering algorithm to achieve this, as we don't have predefined labels for our customer groups.</p>

            <h3>2. Data Pre-processing and Exploration</h3>

            <ul>

                <li>First things first, we brought in our essential Python libraries: <code>numpy</code> for numerical operations, <code>pandas</code> for data manipulation, <code>matplotlib.pyplot</code> and <code>seaborn</code> for creating some nice visualizations, and <code>KMeans</code> from <code>sklearn.cluster</code> for our clustering algorithm.</li>

                <li>We loaded our customer data from the <code>Mall_Customers.csv</code> file into a pandas DataFrame, which we called <code>customer_data</code>.</li>

                <li>Then, we took a good look at our data to understand it. <code>customer_data.head()</code> gave us a quick peek at the first few rows, showing us what kind of information we have. <code>customer_data.shape</code> confirmed the number of rows and columns, giving us a sense of the dataset's size. <code>customer_data.info()</code> was really useful for checking data types and, critically, for seeing if there were any missing values. Thankfully, in this dataset, it looks like there were no null values, which is great – one less thing to worry about! Finally, <code>customer_data.describe()</code> provided a statistical summary of our numerical columns, giving us insights into the range and distribution of age, income, and spending score.</li>

                <li>For this segmentation, we decided to focus on Annual Income (k$) and Spending Score (1-100) because these two features are highly relevant for understanding customer value and behavior. We extracted these two columns into a new variable <code>X</code>.</li>

            </ul>

            <h3>3. Determining Optimal Clusters (Elbow Method)</h3>

            <p>Since we're doing unsupervised learning (meaning we don't have existing labels for our customer segments), we need a way to figure out the "best" number of clusters. The Elbow Method is a common technique for this. We calculated the "Within Cluster Sum of Squares" (WCSS) for a range of possible cluster numbers (from 1 to 10). WCSS essentially measures how compact the clusters are; a lower WCSS generally means better clusters. Then, we plotted the WCSS values against the number of clusters. The idea is to look for an "elbow point" in the graph – where the rate of decrease in WCSS significantly slows down. This point usually indicates the optimal number of clusters, as adding more clusters beyond this point doesn't give you much more compactness. From the plot, the elbow appeared to be at 5 clusters.</p>

            <h3>4. Model Training</h3>

            <p>With the optimal number of clusters identified as 5, we were ready to train our K-Means model. We initialized the <code>KMeans</code> algorithm with <code>n_clusters=5</code>, specifying that we want five distinct customer groups. We also set <code>init='k-means++'</code> (a smart way to pick initial cluster centers that speeds up convergence) and <code>random_state=42</code> for reproducibility (so we get the same results every time we run the code). Then, we trained the model by fitting it to our <code>X</code> data (the Annual Income and Spending Score). After training, the model learned the optimal cluster centers and assigned each data point to one of these five clusters. We stored these cluster assignments in a new column called <code>Cluster</code> in our original <code>customer_data</code> DataFrame.</p>

            <h3>5. Visualization and Insights</h3>

            <p>The best way to understand our customer segments is to visualize them. We used <code>matplotlib.pyplot</code> to create a scatter plot. We plotted Annual Income (k$) on the x-axis and Spending Score (1-100) on the y-axis. Crucially, we colored each data point based on its assigned cluster. This immediately allowed us to see the distinct groups. Looking at the plot, we can now interpret the segments:</p>

            <ul>

                <li><strong>Cluster 0 (e.g., green):</strong> Customers with moderate income but high spending scores. These are likely impulse buyers or trend followers.</li>

                <li><strong>Cluster 1 (e.g., red):</strong> Customers with high income and high spending scores. These are our "prime" or "target" customers – they earn a lot and spend a lot.</li>

                <li><strong>Cluster 2 (e.g., blue):</strong> Customers with low income and low spending scores. These might be budget-conscious shoppers or new customers.</li>

                <li><strong>Cluster 3 (e.g., purple):</strong> Customers with low income but high spending scores. This group is interesting; they might be looking for deals or spending disproportionately to their income.</li>

                <li><strong>Cluster 4 (e.g., orange):</strong> Customers with high income but low spending scores. These are high earners who are cautious spenders at the mall, perhaps saving or spending elsewhere.</li>

            </ul>

            generateInsightButtons.forEach(button => {

                button.addEventListener('click', async function() {

                    const projectDescriptionElement = this.parentElement.querySelector('.project-description');

                    const insightOutputDiv = this.parentElement.querySelector('.insight-text');

                    const originalButtonText = this.innerHTML;


                    insightOutputDiv.innerHTML = '<span class="loading-spinner"></span> Generating insight...';

                    this.disabled = true;


                    const prompt = `Based on the following project description, generate a concise, impactful business insight or key takeaway (1-2 sentences) that highlights the project's value for a business analyst role:\n\n"${projectDescriptionElement.textContent}"`;


                    let chatHistory = [];

                    chatHistory.push({ role: "user", parts: [{ text: prompt }] });

                    const payload = { contents: chatHistory };

                    const apiKey = ""; // Leave as-is. Canvas will provide it.

                    const apiUrl = `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash-preview-05-20:generateContent?key=${apiKey}`;


                    try {

                        let response;

                        let retryCount = 0;

                        const maxRetries = 3;

                        const initialDelay = 1000; // 1 second


                        while (retryCount < maxRetries) {

                            response = await fetch(apiUrl, {

                                method: 'POST',

                                headers: { 'Content-Type': 'application/json' },

                                body: JSON.stringify(payload)

                            });


                            if (response.ok) {

                                break; // Success, exit loop

                            } else if (response.status === 429) { // Too Many Requests

                                const delay = initialDelay * Math.pow(2, retryCount);

                                await new Promise(resolve => setTimeout(resolve, delay));

                                retryCount++;

                            } else {

                                throw new Error(`API error: ${response.status} ${response.statusText}`);

                            }

                        }


                        if (!response.ok) {

                            throw new Error('Failed to fetch from API after multiple retries.');

                        }


                        const result = await response.json();

                        if (result.candidates && result.candidates.length > 0 &&

                            result.candidates[0].content && result.candidates[0].content.parts &&

                            result.candidates[0].content.parts.length > 0) {

                            const text = result.candidates[0].content.parts[0].text;

                            insightOutputDiv.innerHTML = text;

                        } else {

                            insightOutputDiv.innerHTML = 'Could not generate insight. Please try again.';

                        }

                    } catch (error) {

                        insightOutputDiv.innerHTML = `Error: ${error.message}`;

                    } finally {

                        this.disabled = false;

                        this.innerHTML = originalButtonText;

                    }

                });
